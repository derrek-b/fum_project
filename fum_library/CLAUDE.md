# CLAUDE.md — fum_library (Shared Utilities)

## What This Project Is

Shared library consumed by both `fum` (frontend) and `fum_automation` (automation service). Provides platform adapters, blockchain utilities, token/chain configs, contract ABIs, and helper functions. Installed via local tarball — changes aren't visible to consumers until you run `npm run pack`.

## Commands

```bash
npm run build            # Build src/ → dist/ (simple file copy)
npm run pack             # Build + pack tarball + install into fum & fum_automation
npm test                 # Vitest unit tests
npm run test:unit        # Unit tests only
npm run test:integration # Integration tests (requires Hardhat node)
npm run test:coverage    # Coverage report
```

**After any code change**: Always run `npm run pack` before testing in fum or fum_automation. They won't see your changes otherwise.

**Do NOT use npm link (sync/unsync)** to share fum_library with other projects. It causes issues with how the library is initialized across the ecosystem. Always use `npm run pack` for local development.

## Module Structure

```
src/
├── adapters/       # Platform adapters (PlatformAdapter base, UniswapV3, V4, TraderJoeV2_2)
├── artifacts/      # Contract ABIs and deployment addresses (auto-generated by fum/scripts/)
├── blockchain/     # Web3 provider creation, wallet connection, contract instantiation
├── configs/        # Chain configs (chains.js), token lists, platform metadata
├── helpers/        # Utilities: format, chain, token, platform, strategy, Permit2
├── services/       # External APIs: CoinGecko prices, Merkl incentives, The Graph, block explorers
└── index.js        # Main exports (re-exports from all modules)
```

All modules are available as subpath exports:
```js
import { UniswapV3Adapter } from 'fum_library/adapters';
import { getChainConfig } from 'fum_library/helpers/chainHelpers';
import { fetchTokenPrices } from 'fum_library/services/coingecko';
import contracts from 'fum_library/artifacts/contracts';
```

## Adapter Pattern

All DEX interactions go through adapters extending `PlatformAdapter`. Current adapters: UniswapV3, UniswapV4, TraderJoeV2_2. See [adapters.md](docs/architecture/adapters.md) for the full interface, data shapes, and how to add a new platform.

## Build System

The build is a simple file copy (`cp -r src/* dist/`), not a bundler. Source and dist are identical JS files. The `dist/` directory is what gets packed into the tarball and consumed by other projects.

## Test Infrastructure

The library exports test utilities for consumers:
- `fum_library/test/setup/hardhat-config` — Shared Hardhat config and test accounts
- `fum_library/test/setup/test-contracts` — Contract deployment helpers for test environments
- `fum_library/test/test-env` — Test environment setup

These are used by fum_automation's test suite to spin up consistent test environments.

### Adapter test gotchas

- **Port must be 8545.** V3 and V4 adapters create internal AlphaRouter providers using `getChainRpcUrls(1337)` which returns `http://localhost:8545`. The test's Hardhat node must run on that same port or the AlphaRouter will connect to the wrong (or no) node.
- **V4 AlphaRouter needs fresh adapter before late-running tests.** After many `evm_revert` cycles, the AlphaRouter's `JsonRpcProvider` accumulates stale internal block state that causes "invalid block tag N+1" errors. Fix: add `beforeAll(() => { adapter = new UniswapV4Adapter(1337); })` before describe blocks that call `getBestSwapQuote`, `batchSwapTransactions`, or any method that invokes `alphaRouter.route()`. V3 is not affected due to test ordering.
- **`vi.mock` must be at module scope.** Vitest hoists `vi.mock()` calls to the top of the file regardless of where they appear. Placing them inside a test case or describe block causes them to affect all tests in the file. Use module-level `vi.mock` with conditional logic for test-specific chains, or use `vi.spyOn` for test-scoped mocking.

## Key Details

- **ethers.js v5** is a peer dependency — all blockchain code uses v5 API
- Contract ABIs in `src/artifacts/` are auto-generated by `fum/scripts/extract-abis.js` — do not edit manually
- `bytecode/` contains compiled contract bytecodes synced from fum_testing — do not edit manually
- Token lists and chain configs in `src/configs/` are the source of truth for supported tokens and chains
